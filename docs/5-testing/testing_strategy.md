# Testing strategy

justext4 is a filesystem builder — its output gets consumed by Linux kernels and `e2fsck`. Tests fall into four layers, each catching a different bug class.

## Layer 1 — Unit tests (`#[cfg(test)]` per module)

Per-crate, per-module unit tests covering byte layouts, allocator decisions, and individual filesystem ops. Run via:

```sh
cargo test --workspace
```

Convention: every test follows `test_<action>_<condition>_<expectation>` naming and includes a `/// Bug it catches:` doc comment naming the regression class. Example from `spec/src/superblock.rs`:

```rust
/// Bug it catches: a superblock encoder that fails to set the
/// EXT4_FEATURE_INCOMPAT_EXTENTS flag would produce an image whose
/// extent trees the kernel refuses to walk, surfacing as
/// EUCLEAN on first read.
#[test]
fn test_encode_superblock_sets_extents_feature_flag() { /* ... */ }
```

The bug-class doc comment forces test authors to name a *concrete* regression — tests that can't articulate one are noise and get rejected at review.

## Layer 2 — Integration tests (`tests/` per crate)

Full-stack scenarios that exercise multiple modules together: format an image, write some files, close the stream, re-open, read back. These use `Cursor<Vec<u8>>` so they don't touch the filesystem and run in CI without permissions concerns.

## Layer 3 — Kernel-mountability skip-pass tests

The load-bearing differentiator. Bytes that *look* like ext4 and bytes that the kernel *actually accepts* are not the same set. Three external validators are wired in:

| Validator | Env var to enable | What it catches |
|---|---|---|
| `e2fsck` (user-space validator) | `JUSTEXT4_E2FSCK=1` | Structural inconsistencies: superblock/group-descriptor mismatches, dangling inodes, bad block counts. Catches "we wrote a value the spec allows but contradicts another field". |
| `mount -o loop` (Linux kernel) | `JUSTEXT4_MOUNT=1` | What the kernel's actual ext4 driver rejects. Catches "passes e2fsck but a v6.x kernel refuses". |
| `dumpe2fs` (descriptor walk) | (always) | Confirms our descriptor encoding round-trips through user-space tooling. |

**Skip-pass pattern**: tests are *always-on* (no `#[ignore]`). When the env var is unset they print `SKIP: ...` and return early; when set they perform the live check and `panic!` on failure.

```rust
/// Bug it catches: a feature-flag drift between justext4's writer
/// and the kernel's expected ext4 dialect. Unit tests pin the
/// byte layout; only a real kernel mount confirms the kernel
/// agrees.
#[test]
fn test_format_then_mount_loop_succeeds_when_kernel_available() {
    let Some(_kernel) = require_linux_loop_mount() else {
        eprintln!("SKIP: JUSTEXT4_MOUNT != 1");
        return;
    };
    // ...
}
```

## Layer 4 — Fuzzing harness

The `fuzz/` directory (cargo-fuzz) hosts targets that drive the parsers with libFuzzer-generated bytes. Targets cover the high-risk paths:

- Superblock decode (untrusted input)
- Group descriptor decode
- Inode decode (extent tree walk is recursive and easy to break)
- Directory entry decode

Run:

```sh
cargo install cargo-fuzz --locked
cargo +nightly fuzz run superblock_decode -- -max_total_time=300
```

CI exposes a manual-dispatch workflow that runs each target for 5 minutes (`workflow_dispatch` only — fuzzing is expensive).

If the `fuzz/` directory does not yet exist in this repo, treat its absence as a tracked gap; the relevant issue is in the v0-gap label.

## Coverage map

Run `cargo test --workspace 2>&1 | grep "test result:"` to read the current per-crate counts. Numbers shift release-to-release; this section deliberately does not pin specific counts so it doesn't go stale on every PR.

What matters for stability: every public function in `Filesystem` has at least one happy-path test and at least one negative test (file-not-found, no-space, invalid-mode). Coverage holes are tracked as labeled GitHub issues.

## What's NOT tested

Honest gaps:

- **Power-loss / torn-write recovery**: justext4 v0 is read-only-after-format on the data path; the journal is not replayed. Consumers using justext4 as a write-able mounted filesystem on production hardware should not.
- **Concurrent open**: a `Filesystem<R>` is `!Sync`. Sharing across threads is undefined.
- **Encryption / fscrypt**: not implemented; an encrypted ext4 image will fail to parse.
- **Quotas + project quotas**: not implemented.
- **Inline data extents** (small files stored in the inode itself): partially implemented; see the v0-gap labels for status.

Each gap is a labeled GitHub issue under the `v0-gap` label.

## Six-branch flow + CI

Tests run on every push to all six branches (`dev → test → int → uat → prd → main`) via `.github/workflows/ci.yml`. The workflow runs `cargo build` + `cargo test --workspace` + `cargo clippy --workspace --all-targets -- -D warnings` + `cargo fmt --all -- --check` against stable Rust on Ubuntu. The MSRV gate (1.88) runs on the same matrix.

A change must pass on `dev` before it ff-cascades to `test`, and so on through `prd → main`. The `main` branch is what mdBook publishes from.

## Adding a new test

Follow the bug-class convention from layer 1. The naming, the doc comment, and the assertion must all together communicate "this catches the regression class X". A test that just exercises code without naming what it would catch on regression is rejected at review — tests are signal, not noise.
